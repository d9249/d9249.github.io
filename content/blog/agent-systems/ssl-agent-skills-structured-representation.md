에이전트가 실제 업무를 처리하기 시작하면서 병목은 더 이상 모델이 한 번의 답변을 얼마나 그럴듯하게 쓰느냐에만 있지 않다. 오히려 중요한 것은 어떤 작업 지식을 재사용 가능한 형태로 저장하고, 필요할 때 정확히 찾아오며, 실행 전에 위험을 얼마나 빠르게 검토할 수 있느냐다. 최근의 `SKILL.md` 생태계는 이런 문제를 풀기 위한 실용적 해법으로 빠르게 확산됐지만, 동시에 한계도 분명해졌다. 사람이 읽기에는 편한 장문 문서가 기계 입장에서는 invocation 조건, 실행 단계, 실제 부작용이 한데 섞인 비정형 텍스트이기 때문이다.

Peking University의 **"From Skill Text to Skill Structure: The Scheduling-Structural-Logical Representation for Agent Skills"**는 바로 이 지점을 겨냥한다. 이 논문은 스킬을 단순 문서로 보지 않고, 검색 가능한 인터페이스 신호와 실행 구조, 그리고 행위 단위의 자원 사용 흔적까지 분리된 계층으로 정규화해야 한다고 주장한다. 핵심 제안은 이름 그대로 **Scheduling-Structural-Logical, SSL** 표현이다. 논문이 흥미로운 이유는 "스킬을 더 잘 쓰는 프롬프트"를 제안하는 데서 멈추지 않고, 스킬 자체를 기계가 다루기 쉬운 중간 표현으로 재구성하려 한다는 점이다.

에이전트 운영 관점에서 보면 이 문제는 꽤 실무적이다. 스킬 레지스트리가 커질수록 검색은 metadata 몇 줄만으로는 부족해지고, 외부 스킬을 설치하려 할수록 데이터 유출·권한 상승·파괴적 동작 같은 리스크를 사전에 읽어내야 한다. SSL은 이 두 문제를 같은 표현 계층에서 풀려는 시도다. 즉 "좋은 스킬 문서를 쓰자"가 아니라, **스킬 문서에서 기계가 재사용 가능한 구조를 뽑아내자**는 제안에 가깝다.

## 무엇을 해결하려는가

오늘의 스킬 생태계는 빠르게 커지고 있지만, 표현 형식은 여전히 텍스트 중심이다. `SKILL.md`에는 목표, 사용 시점, 절차, 제약, 도구 호출, 파일 변경, 검증 방식이 한 문서 안에 섞여 들어간다. 사람에게는 유연한 형식이지만, 에이전트 런타임이나 스킬 검색기, 보안 검토기 입장에서는 필요한 정보를 매번 다시 추론해야 한다.

논문은 이 상황을 "representation problem"으로 본다. 스킬 검색에서는 단순 제목이나 짧은 설명만으로는 어떤 작업 흐름과 제약을 가진 스킬인지 파악하기 어렵고, 위험 검토에서는 같은 문서 안에서 읽기 전용 동작과 실제 쓰기/전송/권한 사용 동작을 분리하기가 어렵다. 결국 스킬을 많이 모을수록 오히려 **발견 비용과 검토 비용이 함께 증가**한다.

특히 지금의 agent skills 흐름을 생각하면 이 문제는 더 커질 가능성이 높다. 공개 스킬 마켓플레이스, 팀 내부 skill repo, 플러그인 디렉터리, 문서 번들까지 계속 늘어나고 있는데, 이들을 모두 장문 자연어 상태로만 보관하면 검색 품질과 거버넌스 품질이 함께 병목에 걸릴 수밖에 없다. SSL은 이 병목을 "스킬의 의미를 더 작은 구조적 단위로 분해하는 것"으로 풀려 한다.

## 핵심 아이디어 / 구조 / 동작 방식

SSL의 중심 아이디어는 스킬을 세 층으로 나누는 것이다.

- **Scheduling layer**: 이 스킬이 언제 호출되어야 하는지, 어떤 입력과 출력을 기대하는지, 어떤 의존성과 제어 흐름 특성이 있는지를 담는다.
- **Structural layer**: 스킬을 장면(scene) 단위의 실행 흐름으로 표현한다. 준비, 획득, 추론, 행동, 검증, 복구, 종료 같은 매크로 단계가 여기 들어간다.
- **Logical layer**: 실제 원자적 동작을 기록한다. 읽기, 비교, 검증, 쓰기, 도구 호출, 전송, 통지 같은 행위와 그때 접근하는 자원 범위를 명시한다.

이 분해가 중요한 이유는 서로 다른 downstream task가 보는 정보가 다르기 때문이다. 검색기는 주로 Scheduling과 Structural 신호를 원하고, 위험 검토기는 Logical layer의 자원 범위와 효과를 더 강하게 본다. SSL은 하나의 문서 표현으로 이 두 사용 사례를 동시에 겨냥한다.

논문은 이 구조를 사람이 수작업으로 쓰는 포맷으로만 제안하지 않는다. 기존 `SKILL.md`를 **LLM 기반 normalizer**가 SSL JSON 그래프로 바꾸는 4-pass 파이프라인도 함께 제시한다. 공개된 설명에 따르면 이 파이프라인은

1. 스킬 레코드 추출,
2. scene 단위 분해,
3. logic-step 확장,
4. 계층 정합성 검증

순으로 동작한다. 마지막 단계에서 enum, 포인터, 전이 규칙, 포함 관계를 검증하고 malformed output은 재시도한다는 점도 눈에 띈다. 즉 단순 요약이 아니라, **정규화된 구조 표현을 생성하고 검증하는 변환기**로 설계돼 있다.

SSL 스키마의 핵심 필드를 보면 논문의 문제의식이 더 선명해진다.

| 레이어     | 핵심 필드                                                                                                        | 실무적 의미                                                    |
| ---------- | ---------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| Scheduling | `skill_goal`, `intent_signature`, `expected_inputs`, `expected_outputs`, `dependencies`, `control_flow_features` | 스킬 검색과 라우팅에 필요한 인터페이스 표면을 분리             |
| Structural | `scene_id`, `scene_type`, `scene_goal`, `entry_conditions`, `exit_conditions`, `next_scene_rules`                | 멀티스텝 워크플로우를 실행 단계 단위로 해석 가능하게 만듦      |
| Logical    | `act_type`, `instrument`, `resource_scope`, `resource_target`, `effects`, `next_step_rules`                      | 실제 도구 호출과 자원 사용 흔적을 위험 검토 가능한 단위로 기록 |

또한 논문은 allowed vocabulary도 제한한다. 예를 들어 `scene_type`은 `PREPARE`, `ACQUIRE`, `REASON`, `ACT`, `VERIFY`, `RECOVER`, `FINALIZE`로 제한되고, `act_type` 역시 `READ`, `COMPARE`, `VALIDATE`, `WRITE`, `CALL_TOOL`, `TRANSFER`, `NOTIFY` 등으로 고정된다. 이 제한된 어휘는 표현력을 일부 희생하는 대신, 검색과 분석 파이프라인에서 훨씬 다루기 쉬운 신호를 제공한다.

## 공개된 근거에서 확인되는 점

논문은 SSL을 두 가지 태스크로 평가한다. 첫째는 **Skill Discovery**, 둘째는 **Risk Assessment**다. Skill Discovery 실험은 6,184개 스킬 코퍼스와 431개 intent-level query 위에서 수행됐고, Risk Assessment는 252개 스킬 벤치마크에서 여섯 가지 위협 차원을 평가했다.

가장 눈에 띄는 결과는 검색 성능이다. 같은 embedding 모델과 FAISS ranking pipeline을 사용하고, 오직 스킬 표현만 바꿨을 때 `Desc + SSL-Rich`는 가장 좋은 retrieval 성능을 보였다.

| Skill Discovery 입력 표현 |    MRR@50 |    NDCG@5 |   NDCG@10 | Recall@10 |
| ------------------------- | --------: | --------: | --------: | --------: |
| Desc_only                 |     0.588 |     0.608 |     0.626 |     0.761 |
| Full SKILL.md             |     0.645 |     0.655 |     0.682 |     0.821 |
| Desc + Source Outline     |     0.649 |     0.670 |     0.689 |     0.833 |
| Desc + SSL-Shallow        |     0.716 |     0.730 |     0.752 |     0.879 |
| Desc + SSL-Sched          |     0.694 |     0.716 |     0.733 |     0.868 |
| **Desc + SSL-Rich**       | **0.729** | **0.748** | **0.770** | **0.905** |

논문 초록이 강조하듯 MRR@50은 `0.649 → 0.729`로 개선된다. 여기서 중요한 것은 "더 큰 검색 모델을 썼다"가 아니라, **같은 검색기라도 더 구조화된 스킬 표현을 주면 성능이 오른다**는 점이다. 즉 정보 검색 측면에서 SSL은 단순한 문서 정리 포맷이 아니라 실제로 검색기의 입력 품질을 바꾸는 표현 계층으로 작동한다.

Risk Assessment 결과도 의미가 있다. 논문은 데이터 유출, 파괴적 동작, 권한 상승, 은닉 실행, 리소스 남용, 자격 증명 접근 여섯 차원을 평가했는데, `SKILL.md + SSL` 조합이 가장 높은 macro F1을 기록했다.

| Risk Assessment 입력 표현 | Macro Acc. | Macro Prec. | Macro Rec. |  Macro F1 |
| ------------------------- | ---------: | ----------: | ---------: | --------: |
| Desc Only                 |      0.714 |       0.823 |      0.148 |     0.235 |
| Full MD                   |      0.765 |       0.828 |      0.283 |     0.409 |
| SSL-Shallow               |      0.709 |       0.783 |      0.129 |     0.210 |
| Full SSL                  |      0.778 |       0.841 |      0.315 |     0.422 |
| **MD + SSL**              |  **0.801** |   **0.884** |  **0.382** | **0.509** |

세부 차원에서도 improvement가 보인다. 예를 들어 Data Exfiltration은 `Full MD 0.511 → MD + SSL 0.699`, Credential Access는 `0.695 → 0.780`으로 오른다. 반대로 SSL만 단독으로 모든 걸 대체하는 것은 아니다. 논문의 최종 최고 성능은 원문 `SKILL.md`와 SSL을 **함께** 볼 때 나온다. 이 점은 SSL이 원문을 지우는 표준이라기보다, 원문 위에 덧대는 **보조 구조 레이어**라는 해석에 더 가깝다.

공개된 릴리스 성숙도 신호도 살펴볼 만하다. 논문은 GitHub 저장소 `COOLPKU/SSL`을 연결하지만, 조회 시점 기준 이 저장소는 공개 상태이되 README, 릴리스, 태그가 확인되지 않았고 GitHub API 메타데이터상 `size: 0`으로 보인다. 즉 아이디어와 실험 결과는 논문에 분명히 제시되어 있지만, 구현 배포 표면은 아직 매우 초기 단계로 읽는 편이 맞다. 다시 말해 **표현 개념과 실험은 앞서가지만, 운영 가능한 공개 툴링은 아직 얇다**.

## 실무 관점에서의 해석

내가 보기에 이 논문의 진짜 의미는 스킬을 더 많이 모으는 문제가 아니라, **스킬을 어떻게 표현해야 검색과 거버넌스가 가능해지는가**를 본격적으로 다뤘다는 데 있다. 그동안 agent skills 논의는 주로 포맷 채택, 설치 경로, 마켓플레이스, 실행 UX에 집중돼 있었다. 그런데 레지스트리가 커질수록 더 중요한 것은 "이 스킬이 정확히 무슨 일을 하고, 어떤 자원을 건드리며, 언제 선택되어야 하는가"를 기계가 읽을 수 있는 형태로 갖추는 일이다. SSL은 그 missing middle layer를 제안한다.

특히 흥미로운 점은 검색과 보안 검토를 같은 구조 표현으로 묶었다는 것이다. 보통 이 둘은 별도 시스템으로 취급되지만, 실제로는 둘 다 스킬의 인터페이스와 행위 증거를 더 잘 분리해서 보고 싶어 한다. Scheduling/Structural이 discovery에 도움을 주고, Logical이 risk review에 도움을 준다는 실험 결과는 꽤 설득력 있다. 향후 스킬 마켓플레이스나 기업 내부 skill registry가 커질수록 이런 구조화 계층은 점점 더 중요해질 가능성이 높다.

동시에 한계도 분명하다. 첫째, 제한된 vocabulary는 분석 가능성을 높이는 대신 실제 복잡한 스킬의 nuance를 잘라낼 수 있다. 둘째, LLM normalizer가 만든 구조가 얼마나 안정적으로 source-grounded한지는 장기적으로 더 넓은 human audit이 필요하다. 셋째, 현재 공개 저장소 성숙도를 보면 SSL이 바로 생산 환경의 사실상 표준으로 들어온 단계는 아니다. 지금은 잘 정리된 운영 시스템이라기보다, **표현 계층의 필요성을 수치로 보여준 연구 프로토타입**에 더 가깝다.

그럼에도 방향성은 꽤 중요하다. 에이전트 시대의 다음 경쟁력은 더 긴 프롬프트나 더 큰 모델만이 아니라, 작업 지식을 얼마나 구조적으로 저장하고 검색하고 검토할 수 있느냐에 달려 있을 수 있다. 그런 의미에서 SSL은 또 하나의 `SKILL.md` 변형이라기보다, **agent skills를 문서 자산에서 운영 가능한 지식 그래프 쪽으로 밀어붙이는 초기 신호**로 보는 편이 더 맞다.

특히 스킬 레지스트리를 운영하거나, 외부 스킬을 사내 환경에 들이기 전에 자동 사전 검토가 필요한 팀이라면 이 논문은 꽤 직접적인 힌트를 준다. 앞으로의 스킬 시스템은 "문서가 있느냐"보다 "문서에서 구조를 얼마나 안정적으로 추출할 수 있느냐"로 평가받게 될 가능성이 크기 때문이다.
